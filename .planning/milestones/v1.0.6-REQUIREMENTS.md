# Requirements Archive: v1.0.6

**Archived:** 2026-04-12
**Core Value:** Provides immediate, actionable visibility into model quotas to prevent unexpected usage limits.

## v1 Requirements (Milestone v1.0.6)

### API Reliability
- [x] **RETRY-01**: Detect "High Traffic" error via HTTP 429 status code. (Validated: Phase 1)
- [x] **RETRY-02**: Detect "High Traffic" error via text match in response body. (Validated: Phase 1)
- [x] **RETRY-03**: Implement exponential backoff (2^n * 1000ms). (Validated: Phase 2)
- [x] **RETRY-04**: Limit retry attempts to exactly 5. (Validated: Phase 2)

### UI / UX
- [x] **UI-01**: Display countdown/progress indicator when retry is in progress. (Validated: Phase 3)
- [x] **UI-02**: Show distinct "Retrying..." state in Status Bar. (Validated: Phase 3)
- [x] **UI-03**: Allow user to see how many retry attempts have been made. (Validated: Phase 3)
- [x] **UI-06-01**: Clickable Status Bar icon to toggle retry feature. (Validated: Phase 6)
- [x] **UI-06-02**: Toggle in sidebar Webview to enable/disable retry feature. (Validated: Phase 6)

### Configuration
- [x] **CONF-02**: Allow user to disable auto-retry. (Validated: Phase 5)

## Out of Scope
- Retrying 401/403 errors (require auth refresh).
- Global Rate Limiting (handled by server).
